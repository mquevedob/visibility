# v0.1 Roadmap

## Roadmap intent

This roadmap turns the MVP definition and architecture into small implementation phases.

Each phase should be small enough to become one Codex implementation prompt.

v0.1 analyzes **one product at a time**.

Batch analysis, scheduled monitoring, dashboards, live search-result acquisition, AI visibility checks, and semantic/vector matching are explicitly out of scope.

## v0.1 product boundary

The core API should eventually support this shape:

```php
VisibilityAnalyzer::analyze(
    ProductSubject $product,
    array $queries,
    array $resultSets
): VisibilityReport;
```

Where:

- `$product` is one product.
- `$queries` contains one or more query contexts for that product.
- `$resultSets` contains observed search result snapshots supplied from outside the core.
- The analyzer returns one `VisibilityReport` for that product.

There is no `BatchVisibilityAnalyzer` in v0.1.

## v0.1 non-goals

Do not implement these in v0.1:

- live Google scraping;
- live Bing scraping;
- marketplace scraping;
- AI answer-engine checks;
- embeddings;
- vector databases;
- semantic product-query matching;
- browser rendering;
- full-site crawling;
- Laravel integration;
- dashboard UI;
- SaaS account/project model;
- scheduled monitoring;
- competitor reports;
- batch analysis for many products.

## Phase 1 ŌĆö Package skeleton

### Deliverables

- `tests/`


## Phase 2 ŌĆö Core value objects

### Goal

Implement immutable value objects for the core evidence and report model.

### Deliverables

- `ProductSubject`
- `SearchQuery`
- `SearchResult`
- `SearchResultSet`
- `UrlMatch`
- `PageSnapshot`
- `ParsedPage`
- `Finding`
- `QueryVisibility`
- `VisibilityReport`

### Acceptance criteria

- Value objects can be constructed from arrays or explicit constructors.
- Required fields are validated.
- Optional fields have safe defaults.
- Objects can be converted to arrays for JSON serialization.
- Tests cover valid and invalid construction.

### Design notes

Do not overbuild validation. v0.1 needs useful guardrails, not a full schema engine.

`ProductSubject` must represent exactly one product.

It should support:

- one required `expectedUrl`;
- optional `acceptableUrlVariants`;
- optional `expectedTerms`.

### Codex prompt seed

```text
Implement the core value objects defined in docs/architecture.md.

Focus only on immutable data structures and tests.

Rules:
- ProductSubject represents one product only.
- expectedUrl is required.
- acceptableUrlVariants is optional.
- SearchResultSet is supplied evidence, not collected by this package.
- Include toArray() methods for all reportable objects.
- Add PHPUnit tests for construction, defaults, and required-field validation.
- Do not implement analyzer logic yet.
```

## Phase 3 ŌĆö Static search evidence provider

### Goal

Provide deterministic search-result evidence for tests and local demos.

### Deliverables

- `SearchProvider` interface
- `StaticSearchProvider`
- fixture-loading helper or constructor-based fixture input

### Acceptance criteria

- Static provider returns configured `SearchResultSet` objects.
- It does not call external services.
- It can return multiple result sets for one product and multiple queries.
- Tests cover one query, multiple queries, and empty results.

### Codex prompt seed

```text
Implement SearchProvider and StaticSearchProvider.

The provider must return preconfigured SearchResultSet objects.
It must not call external APIs.
It must be suitable for deterministic tests.

Add tests for:
- one query with results
- one query with no results
- multiple queries
- provider warnings/limitations passthrough
```

## Phase 4 ŌĆö URL normalization and matching

### Goal

Detect whether the expected product URL appears in observed results.

### Deliverables

- `UrlNormalizer`
- `UrlMatcher` interface
- `DefaultUrlMatcher`
- `UrlMatch` result handling

### Acceptance criteria

The matcher detects:

- exact match;
- normalized host/scheme match;
- trailing slash equivalence;
- fragment-insensitive match;
- tracking-parameter-insensitive match;
- acceptable variant match;
- no match.

### Initial tracking parameters

At minimum strip:

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_term`
- `utm_content`
- `gclid`
- `fbclid`

### Codex prompt seed

```text
Implement URL normalization and matching.

Requirements:
- UrlNormalizer normalizes scheme/host casing.
- It strips fragments.
- It handles trailing slash equivalence.
- It removes common tracking parameters.
- DefaultUrlMatcher compares ProductSubject.expectedUrl and acceptableUrlVariants against SearchResultSet results.
- Return UrlMatch with matched flag, matchType, matched URL, matched position, and evidence.

Add PHPUnit tests for exact, normalized, tracking-param, fragment, acceptable-variant, and no-match cases.
```

## Phase 5 ŌĆö Visibility result detector

### Goal

Generate query-level visibility status from URL matching evidence.

### Deliverables

- `DetectionContext`
- `Detector` interface
- `VisibilityResultDetector`

### Finding codes

- `product.visible_in_results`
- `product.not_found_in_results`
- `product.visibility_uncertain`

### Acceptance criteria

- If URL match exists, status is `visible`.
- If no URL match exists and results are available, status is `not_visible`.
- If provider warnings/limitations make evidence incomplete, status can be `uncertain`.
- Findings include structured evidence.

### Codex prompt seed

```text
Implement DetectionContext, Detector, and VisibilityResultDetector.

Use UrlMatch and SearchResultSet evidence to generate query visibility findings.

Rules:
- matched URL => visible
- no match with usable result set => not_visible
- provider limitation or incomplete result set can produce uncertain
- findings must include code, severity, confidence, evidence, and recommendation

Add tests for visible, not_visible, and uncertain.
```

### Acceptance criteria

- Tests cover 200 HTML, 404, redirect, timeout/failure, and non-HTML response.

### Codex prompt seed

```text
Implement PageFetcher and FixturePageFetcher.

The fixture fetcher must return PageSnapshot objects from configured fixtures.
No real HTTP calls.

Add tests for:
- 200 HTML response
- 404 response
- redirect evidence
- timeout/failure evidence
- missing fixture
- non-HTML content type
```

## Phase 7 ŌĆö DOM page parser

### Goal

Extract useful product-page evidence from HTML and headers.

### Deliverables

- `PageParser` interface
- `DomPageParser`
- JSON-LD extraction helper
- parser warnings

### Extracted fields

- title
- meta description
- canonical URL
- meta robots directives
- X-Robots-Tag headers
- hreflang links
- H1/H2 headings
- basic links
- JSON-LD blocks
- schema.org types
- Product candidates
- Offer candidates
- body text summary

### Acceptance criteria

- Parser handles valid HTML.
- Parser handles missing fields.
- Parser handles malformed JSON-LD with warnings.
- Parser does not throw for common bad HTML.
- Tests use raw HTML/header fixtures.

### Codex prompt seed

```text
Implement PageParser and DomPageParser.

Extract title, meta description, canonical, robots directives, X-Robots-Tag, hreflang, H1/H2, links, JSON-LD blocks, schema.org types, Product candidates, Offer candidates, and body text summary.

Rules:
- tolerate malformed HTML
- malformed JSON-LD should create parser warnings, not fatal errors
- no browser rendering
- no external network calls

Add fixture-based tests for normal product page, missing metadata, noindex, canonical, JSON-LD Product/Offer, and malformed JSON-LD.
```

## Phase 8 ŌĆö HTTP availability detector

### Deliverables

- `page.http_error`
- `page.redirects_elsewhere`
- `page.non_html_response`

### Acceptance criteria

- Non-2xx status produces finding.
- Fetch failure produces finding.
- Redirect to unrelated URL produces finding.
- Non-HTML content type produces finding.
- Findings include evidence.

### Codex prompt seed

```text
Implement HttpAvailabilityDetector.

Use PageSnapshot evidence to detect fetch failure, HTTP error, redirects elsewhere, and non-HTML responses.

Add tests for each finding code and a clean 200 HTML case.
```

## Phase 9 ŌĆö Indexability detector

### Goal

Detect indexability blockers.

### Deliverables

- `IndexabilityDetector`

### Finding codes

- `indexability.noindex_meta`
- `indexability.noindex_x_robots_tag`
- `indexability.robots_blocked`
- `indexability.unavailable_after_expired`

### Acceptance criteria

- Meta robots `noindex` produces finding.
- X-Robots-Tag `noindex` produces finding.
- `none` is treated as noindex/nofollow evidence.
- Expired `unavailable_after` produces finding.
- robots.txt evidence can be represented if present, but live robots fetching is not required.

### Codex prompt seed

```text
Implement IndexabilityDetector.

Use ParsedPage and PageSnapshot evidence.
Detect meta robots noindex/none, X-Robots-Tag noindex, expired unavailable_after, and optional robots-blocked evidence if present.

Do not fetch robots.txt live.
Add tests for each directive and a clean indexable page.
```

## Phase 10 ŌĆö Canonical detector

### Goal

Detect canonical URL problems that can cause a product page to lose visibility.

### Deliverables

- `CanonicalDetector`

- `canonical.missing`
- `canonical.invalid`
- `canonical.relative`
- `canonical.multiple_conflicting`
- `canonical.points_to_other_url`
- `canonical.points_to_homepage`

### Acceptance criteria

- Missing canonical can produce low/medium severity finding.
- Invalid canonical produces finding.
- Relative canonical produces finding.
- Canonical pointing to a different product/category/homepage produces finding.
- Canonical matching expected URL is clean.

### Codex prompt seed

```text
Implement CanonicalDetector.

Use ProductSubject.expectedUrl, acceptableUrlVariants, ParsedPage.canonicalUrl, and URL normalization.
Detect missing, invalid, relative, conflicting, other URL, and homepage canonical cases.

Add tests for clean canonical and each finding code.
```

## Phase 11 ŌĆö Structured data detector

### Goal

Detect missing or incomplete ecommerce structured data.

### Deliverables

- `StructuredDataDetector`

### Finding codes

- `schema.product_missing`
- `schema.product_invalid_jsonld`
- `schema.offer_missing`
- `schema.price_missing`
- `schema.currency_missing`
- `schema.availability_missing`
- `schema.image_missing`
- `schema.identifier_missing`

### Acceptance criteria

- Missing Product schema produces finding.
- Malformed JSON-LD warning produces finding.
- Product without Offer produces finding.
- Offer without price/currency/availability produces findings.
- Missing image produces finding.

### Codex prompt seed

```text

Detect missing Product, malformed JSON-LD, missing Offer, missing price, missing currency, missing availability, missing image, and missing identifiers.

Support @graph when practical.
Keep implementation deterministic and tested with raw JSON-LD fixtures.
```

## Phase 12 ŌĆö Basic content alignment detector

### Goal

Detect simple content/query alignment issues without semantic AI.

### Deliverables

- `ContentAlignmentDetector`

### Finding codes

- `content.title_missing_product_terms`
- `content.h1_missing_product_terms`
- `content.description_missing`
- `content.description_too_thin`
- `content.body_missing_product_terms`

### Acceptance criteria

- Expected terms are compared against title, H1, description, and body summary.
- Matching is case-insensitive.
- Empty expected terms should not produce noisy findings.
- No embeddings, no LLMs.

### Codex prompt seed

```text
Implement ContentAlignmentDetector.

Compare ProductSubject.expectedTerms against ParsedPage title, H1, meta description, and body text summary.
Use simple case-insensitive lexical matching only.

Detect missing title terms, missing H1 terms, missing description, thin description, and body missing product terms.

Add tests for clean alignment, partial alignment, missing description, thin description, and empty expected terms.
```

## Phase 13 ŌĆö Visibility analyzer orchestration

### Goal

Wire providers, matcher, fetcher, parser, and detectors into one product-level report.

### Deliverables

- `VisibilityAnalyzer`
- default analyzer constructor or factory
- query-level `QueryVisibility` assembly
- top-level `VisibilityReport`

### Acceptance criteria

- Analyzer accepts one product, multiple queries, and result sets.
- Analyzer returns one report for that product.
- Analyzer runs URL matching per query.
- Analyzer fetches/parses the product page once per analysis.
- Analyzer runs detectors.
- Analyzer produces stable JSON-serializable output.

### Codex prompt seed

```text
Implement VisibilityAnalyzer orchestration.

Input:
- one ProductSubject
- SearchQuery[]
- SearchResultSet[]

Output:
- one VisibilityReport

Rules:
- one product per analysis
- no batch analyzer
- no external search calls
- fetch and parse product page once
- run URL matching per query
- run detectors and attach findings to QueryVisibility or summary findings as appropriate

Add end-to-end fixture tests for visible, not_visible with canonical issue, noindex issue, and missing schema issue.
```

## Recommended implementation order

Implement in this order:

1. Package skeleton.
2. Core value objects.
3. Static search provider.
4. URL normalization and matching.
5. Visibility result detector.
6. Fixture page fetcher.
7. DOM page parser.
8. HTTP availability detector.
9. Indexability detector.
10. Canonical detector.
11. Structured data detector.
12. Content alignment detector.
13. Visibility analyzer orchestration.
14. JSON serializer.
15. Demo fixtures and README usage.

## v0.1 completion definition

v0.1 is complete when:

- one product can be analyzed end-to-end;
- observed search result snapshots are accepted as input;
- expected product URL matching works;
- product page evidence is fetched from fixtures;
- HTML/page evidence is parsed;
- indexability, canonical, structured data, HTTP, and content findings are generated;
- a JSON visibility report is produced;
- all success criteria from `docs/mvp.md` have fixture tests;
- no external search acquisition is implemented.

- live search provider adapter;
- marketplace adapter;
- rendered page adapter;

- historical storage;
- semantic ProductQueryMatcher;
