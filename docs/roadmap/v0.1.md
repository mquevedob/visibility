# v0.1 Roadmap

## Roadmap intent

This roadmap turns the MVP definition and architecture into small implementation phases.

Each phase should be small enough to become one Codex implementation prompt.

v0.1 analyzes **one product at a time**.

Batch analysis, scheduled monitoring, dashboards, live search-result acquisition, AI visibility checks, and semantic/vector matching are explicitly out of scope.

v0.1 is a deterministic core engine. It does **not** decide automatically which queries a product should appear for. Expected queries, query intent, and query priority are supplied by the caller.

## Progress tracking rules

This roadmap is the source of truth for v0.1 implementation progress.

Every phase must keep this progress block directly under the phase title:

```md
**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —
```

Allowed statuses:

- `Pending` — not started.
- `In Progress` — implementation has started but is incomplete or not fully validated.
- `Implemented` — code has been created, but acceptance criteria are not fully validated yet.
- `Validated` — implementation is complete and tests/acceptance criteria pass.
- `Blocked` — implementation cannot proceed until a dependency, design decision, or bug is resolved.
- `Skipped` — intentionally removed from v0.1 scope.

Progress rules:

- Use `0%` for `Pending`.
- Use an approximate percentage for `In Progress`.
- Use `80%` to `99%` for `Implemented` when tests or acceptance validation are still pending.
- Use `100%` only for `Validated`.
- Update `Implemented in` with commit SHA, branch name, or PR reference when available.
- Update `Validated by` with the exact test command, manual check, or acceptance evidence used.
- Do not mark a phase as `Validated` unless its acceptance criteria pass.
- Do not change the phase scope silently while marking progress. If scope changes, update the phase text explicitly.

Codex and future implementation chats should update these fields whenever a phase is started, implemented, blocked, skipped, or validated.

## v0.1 product boundary

The core API should eventually support this shape:

```php
VisibilityAnalyzer::analyze(
    ProductSubject $product,
    array $queries,
    array $resultSets
): VisibilityReport;
```

Where:

- `$product` is one product.
- `$queries` contains one or more query contexts for that product.
- `$resultSets` contains observed search result snapshots supplied from outside the core.
- The analyzer returns one `VisibilityReport` for that product.

There is no `BatchVisibilityAnalyzer` in v0.1.

## v0.1 non-goals

Do not implement these in v0.1:

- live Google scraping;
- live Bing scraping;
- marketplace scraping;
- AI answer-engine checks;
- embeddings;
- vector databases;
- semantic product-query matching;
- automatic expected-query generation;
- browser rendering;
- full-site crawling;
- Laravel integration;
- dashboard UI;
- SaaS account/project model;
- scheduled monitoring;
- competitor reports;
- batch analysis for many products;
- revenue forecasting or attribution modeling.

## Phase 2 — Core value objects

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** current PR / branch  
**Validated by:** Owner-accepted implementation; runtime validation handled manually after merge

### Goal

Implement immutable value objects for the core evidence, commercial context, and report model.

### Deliverables

- `ProductSubject`
- `SearchQuery`
- `SearchResult`
- `SearchResultSet`
- `UrlMatch`
- `PageSnapshot`
- `ParsedPage`
- `Finding`
- `QueryVisibility`
- `ReportSummary`
- `VisibilityReport`

### Acceptance criteria

- Value objects can be constructed from arrays or explicit constructors.
- Required fields are validated.
- Optional fields have safe defaults.
- Objects can be converted to arrays for JSON serialization.
- Tests cover valid and invalid construction.
- Query intent, query priority, and product commercial context can be represented but are not automatically inferred.

### Design notes

Do not overbuild validation. v0.1 needs useful guardrails, not a full schema engine.

`ProductSubject` must represent exactly one product.

It should support:

- required `expectedUrl`;
- optional `id`;
- optional `name`;
- optional `brand`;
- optional `sku`;
- optional `category`;
- optional `acceptableUrlVariants`;
- optional `expectedTerms`;
- optional `commercialPriority` such as `low`, `medium`, `high`, or `critical`;
- optional `commercialValue` as caller-supplied metadata;
- optional `price`, `currency`, and `stockStatus` as caller-supplied context.

`SearchQuery` should support:

- required `text`;
- required `provider`;
- optional `locale`;
- optional `device`;
- optional `intent`;
- optional `expectedVisibility` boolean;
- optional `priority` such as `low`, `medium`, `high`, or `critical`;
- optional `reason` explaining why the caller expects the product to appear.

v0.1 must only represent these fields. It must not infer query intent, expected visibility, priority, or commercial value automatically.

### Codex prompt seed

```text
Implement the core value objects defined in docs/architecture.md.

Focus only on immutable data structures and tests.

Rules:
- ProductSubject represents one product only.
- expectedUrl is required.
- acceptableUrlVariants is optional.
- expectedTerms is optional.
- ProductSubject may include caller-supplied commercial context: category, commercialPriority, commercialValue, price, currency, and stockStatus.
- SearchQuery must include text and provider.
- SearchQuery may include locale, device, intent, expectedVisibility, priority, and reason.
- Do not infer intent, expectedVisibility, priority, commercialPriority, or commercialValue automatically.
- SearchResultSet is supplied evidence, not collected by this package.
- Include toArray() methods for all reportable objects.
- Add PHPUnit tests for construction, defaults, and required-field validation.
- Do not implement analyzer logic yet.
```

## Phase 5 — Visibility result detector

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** current PR / branch  
**Validated by:** Owner-accepted implementation; runtime validation handled manually after merge

### Goal

Generate query-level visibility status from URL matching evidence.

### Deliverables

- `DetectionContext`
- `Detector` interface
- `VisibilityResultDetector`

### Finding codes

- `product.visible_in_results`
- `product.not_found_in_results`
- `product.visibility_uncertain`
- `query.expected_visibility_missing`

### Acceptance criteria

- If URL match exists, status is `visible`.
- If no URL match exists and results are available, status is `not_visible`.
- If provider warnings/limitations make evidence incomplete, status can be `uncertain`.
- If a query is marked as expected visible and the product is not found, findings should preserve that expected-visibility context.
- Findings include structured evidence.

### Codex prompt seed

```text

Use UrlMatch and SearchResultSet evidence to generate query visibility findings.

Rules:
- matched URL => visible
- no match with usable result set => not_visible
- provider limitation or incomplete result set can produce uncertain
- if SearchQuery.expectedVisibility is true and no match exists, preserve that context in finding evidence
- findings must include code, severity, confidence, evidence, and recommendation

Add tests for visible, not_visible, uncertain, and expected-visible-but-not-found.
```

**Validated by:** —

### Goal

Turn raw findings into a deterministic product-level summary with prioritized next actions.

### Deliverables

- `FindingPrioritizer`
- `ReportSummarizer`
- `ReportSummary`

### Summary fields

The summary should support:

- overall visibility status;
- highest priority affected query;
- top probable causes;
- top recommended actions;
- overall priority such as `low`, `medium`, `high`, or `critical`;
- short human-readable summary message;
- evidence references to the findings that drove the summary.

### Acceptance criteria

- Not visible plus expected-visible query can increase priority.
- Query priority and product commercial priority influence summary priority.
- Technical blockers such as HTTP errors, `noindex`, and canonical pointing elsewhere rank above weak content findings.
- Missing Product/Offer structured data can be summarized as a visibility-quality issue.
- Summary is deterministic and testable.
- No LLM, no revenue forecasting, and no attribution model.

### Codex prompt seed

```text

- no LLMs

- not_visible + expectedVisibility=true should increase priority
- high or critical query/product priority should increase priority
- HTTP errors, noindex, and canonical-to-other-url should rank above weak content findings
- output top probable causes and top recommended actions

Add tests for:
- visible product with low-priority summary
- not_visible expected query with high priority
```

## v0.1 completion definition

v0.1 is complete when:

- one product can be analyzed end-to-end;
- observed search result snapshots are accepted as input;
- expected product URL matching works;
- query expected-visibility context can be represented;
- product commercial context can be represented;
- product page evidence is fetched from fixtures;
- HTML/page evidence is parsed;
- indexability, canonical, structured data, HTTP, and content findings are generated;
- deterministic summary and prioritized next actions are generated;
- a JSON visibility report is produced;
- all success criteria from `docs/mvp.md` have fixture tests;
- no external search acquisition is implemented.
