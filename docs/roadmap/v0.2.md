# v0.2 Roadmap

## Roadmap intent

This roadmap turns the v0.1 follow-up findings into small implementation phases.

Each phase should be small enough to become one Codex implementation prompt.

v0.2 improves diagnostic clarity, URL evidence handling, and local scenario usability for the deterministic one-product analyzer.

v0.2 still analyzes **one product at a time**.

Batch analysis, scheduled monitoring, dashboards, live search-result acquisition, AI visibility checks, semantic/vector matching, Laravel integration, and SaaS workflows remain explicitly out of scope.

## Progress tracking rules

This roadmap is the source of truth for v0.2 implementation progress.

Every phase must keep this progress block directly under the phase title:

```md
**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —
```

Allowed statuses:

- `Pending` — not started.
- `In Progress` — implementation has started but is incomplete or not fully validated.
- `Implemented` — code has been created, but acceptance criteria are not fully validated yet.
- `Validated` — implementation is complete and tests/acceptance criteria pass.
- `Blocked` — implementation cannot proceed until a dependency, design decision, or bug is resolved.
- `Skipped` — intentionally removed from v0.2 scope.

Progress rules:

- Use `0%` for `Pending`.
- Use an approximate percentage for `In Progress`.
- Use `80%` to `99%` for `Implemented` when tests or acceptance validation are still pending.
- Use `100%` only for `Validated`.
- Update `Implemented in` with commit SHA, branch name, or PR reference when available.
- Update `Validated by` with the exact test command, manual check, or acceptance evidence used.
- Do not mark a phase as `Validated` unless its acceptance criteria pass.
- Do not change the phase scope silently while marking progress. If scope changes, update the phase text explicitly.

Codex and future implementation chats should update these fields whenever a phase is started, implemented, blocked, skipped, or validated.

## v0.2 product boundary

The v0.2 core remains centered on this API shape:

```php
VisibilityAnalyzer::analyze(
    ProductSubject $product,
    array $queries
): VisibilityReport;
```

Where:

- `$product` is one product.
- `$queries` contains one or more query contexts for that product.
- Search-result evidence is still supplied from outside the core through a provider.
- Page evidence is still supplied through a fetcher abstraction.
- The analyzer returns one `VisibilityReport` for that product.

There is no `BatchVisibilityAnalyzer` in v0.2.

## v0.2 non-goals

Do not implement these in v0.2:

- live Google scraping;
- live Bing scraping;
- marketplace scraping;
- AI answer-engine checks;
- embeddings;
- vector databases;
- semantic product-query matching;
- automatic expected-query generation;
- browser rendering;
- full-site crawling;
- Laravel integration;
- dashboard UI;
- SaaS account/project model;
- scheduled monitoring;
- competitor reports;
- batch analysis for many products;
- revenue forecasting or attribution modeling.

## Phase 1 — Finding taxonomy cleanup

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** PR #24  
**Validated by:** composer test

### Goal

Reduce duplicate or overlapping findings so reports are easier to read and prioritize.

### Problem

v0.1 can emit conceptually duplicate findings, especially:

- `page.canonical_mismatch` and `canonical.points_to_other_url`;
- `page.product_schema_missing` and `schema.product_missing`.

These are useful signals, but they create noisy summaries and repeated recommendations.

### Deliverables

- A single preferred finding code for canonical URL mismatch.
- A single preferred finding code for missing Product schema.
- A clear decision about whether legacy `page.*` findings are removed, downgraded, or excluded from summaries.
- Updated tests that prove duplicate canonical/schema findings do not appear together in the same query report.
- Updated sample report and example expectations if affected.

### Acceptance criteria

- A canonical mismatch produces one primary canonical finding in the final report.
- Missing Product schema produces one primary Product schema finding in the final report.
- Summary probable causes and recommended actions do not repeat the same root cause under different codes.
- Existing high-value evidence is preserved under the chosen finding code.
- Existing v0.1 behavior for visibility status is not changed.
- Phase 1 does not add `visibilityHealth`, `at_risk`, `blocked`, `healthy`, visible-at-risk summary messaging, or any other health/risk classification. Those belong to Phase 2.
- `composer test` passes after implementation.

### Design notes

Preferred canonical code:

- `canonical.points_to_other_url`

Preferred Product schema code:

- `schema.product_missing`

Offer schema may remain distinct as:

- `schema.offer_missing`

If renaming `page.offer_schema_missing` is too disruptive, keep it for now but document the reason in tests or code comments.

Phase 1 is limited to taxonomy cleanup and duplicate suppression. It must not implement the visibility health layer, risk states, or summary health messaging planned for Phase 2.

## Phase 2 — Visibility health layer

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** PR #25 + PR #26  
**Validated by:** vendor/bin/phpunit

### Goal

Separate result-set visibility from technical discoverability risk.

### Problem

A product can be `visible` in supplied search results while still having serious page risks such as `noindex`, canonical mismatch, or missing Product schema. v0.1 reports this correctly, but the summary can read as if visibility and technical health are the same thing.

### Deliverables

- A report-level or query-level visibility health field.
- Health classification based on findings without changing the existing visibility status.
- Updated summary message generation that distinguishes visible/healthy from visible/at-risk.
- Tests for visible clean, visible at-risk, not visible, and uncertain cases.

### Acceptance criteria

- `status` remains one of `visible`, `not_visible`, or `uncertain`.
- A visible product with high-severity indexability/canonical availability blockers is marked as at risk.
- A visible product without blocker/high-risk findings is marked as healthy or equivalent.
- A not-visible product is not incorrectly described as merely at risk.
- Summary text clearly says when a product is visible but discoverability is at risk.
- `composer test` passes after implementation.

### Design notes

Suggested field names:

```json
{
  "visibilityHealth": "healthy|at_risk|blocked|unknown"
}
```

Alternative names are acceptable if they are consistent and tested.

Health should be derived from findings, not from new external evidence.

## Phase 3 — URL evidence policy and fetch target clarity

**Status:** Implemented  
**Progress:** 90%  
**Implemented in:** local worktree  
**Validated by:** Owner manual validation pending

### Goal

Make URL evidence explicit and predictable across exact matches, normalized matches, acceptable variants, redirects, final URLs, and canonical URLs.

### Problem

v0.1 works, but local examples exposed ambiguity between:

- matched URL from search results;
- requested URL used by the fetcher;
- final URL after fixture/redirect evidence;
- expected product URL;
- canonical URL declared by the page.

### Deliverables

- Documented URL evidence policy in code docs or README.
- Report evidence fields that make `matchedUrl`, `requestedUrl`, `finalUrl`, `expectedUrl`, and `canonicalUrl` easy to distinguish.
- Optional `FixturePageFetcher` normalized fallback, if it can be implemented without hiding important missing-fixture failures.
- Tests for exact URL, tracking-param normalized URL, acceptable variant URL, and redirect/final URL cases.

### Acceptance criteria

- Reports clearly preserve the URL found in the search result as `matchedUrl`.
- Page fetch evidence clearly preserves `requestedUrl` and `finalUrl`.
- Canonical diagnostics compare canonical URL against expected URL plus acceptable variants.
- Normalized and acceptable variant examples do not require misleading fixture duplication.
- Missing fixtures still produce controlled failure evidence.
- `composer test` passes after implementation.

### Design notes

Suggested policy:

```text
matchedUrl = URL found in search-result evidence
requestedUrl = URL sent to PageFetcher
finalUrl = URL after redirects or fixture final URL
expectedUrl = merchant/product expected URL
canonicalUrl = canonical URL declared in parsed page
```

Evaluation policy:

```text
Visibility match = expectedUrl or acceptableUrlVariants compared against matchedUrl
Page diagnostics = requestedUrl/finalUrl/page body evidence
Canonical diagnostics = canonicalUrl compared against expectedUrl + acceptableUrlVariants
```

### Goal

Separate technical page conditions into isolated HTML fixtures so each scenario validates one root cause clearly.

### Problem

### Deliverables

- `examples/fixtures/product-page-canonical-mismatch.html`
- `examples/fixtures/product-page-missing-schema.html`
- Scenario configuration or runner support for choosing the page fixture.
- Tests that each HTML fixture produces the expected focused diagnostic signal.

### Acceptance criteria

- Noindex fixture emits the noindex finding without requiring canonical/schema defects.
- Canonical mismatch fixture emits canonical mismatch without requiring noindex/schema defects.
- Missing schema fixture emits schema missing without requiring noindex/canonical defects.
- Existing examples remain runnable.
- `composer test` passes after implementation.

### Design notes

Keep fixtures small, readable, and deterministic.

## Phase 5 — Minimal scenario CLI

**Status:** Pending  
**Progress:** 0%  
**Implemented in:** —  
**Validated by:** —

### Goal

Provide a minimal developer-facing CLI for running deterministic local scenarios without creating or editing PHP scripts per scenario.

### Problem

v0.1.1 introduced separate PHP scripts per scenario. This is acceptable for manual validation, but it does not scale cleanly as scenario count grows.

### Deliverables

- Composer bin entry, for example `vendor/bin/visibility`.
- `analyze` command that accepts one scenario JSON file.
- Scenario JSON shape that includes product, query, search fixture, and page fixture references.
- Example scenario files under `examples/scenarios/`.
- README documentation for CLI usage.
- Tests for valid scenario execution, invalid file, invalid JSON, and missing fixture references.

### Acceptance criteria

- A user can run one deterministic scenario with a command similar to:

```bash
vendor/bin/visibility analyze examples/scenarios/not-visible.json
```

- CLI output is the existing JSON report format.
- CLI does not call external services.
- CLI does not introduce Laravel, Symfony Console, or heavy framework dependency unless explicitly justified.
- Existing example scripts continue to work or are clearly deprecated with replacements.
- `composer test` passes after implementation.

### Design notes

Prefer a small plain-PHP CLI entrypoint before adding dependencies.

Scenario JSON should be explicit rather than clever.

Suggested scenario shape:

```json
{
  "product": {
    "expectedUrl": "https://example.test/products/aurora-trail-shoe",
    "name": "Aurora Trail Shoe",
    "brand": "Acme Outdoor",
    "acceptableUrlVariants": []
  },
  "queries": [
    {
      "text": "acme waterproof trail running shoes",
      "provider": "static-fixture",
      "expectedVisibility": true,
      "priority": "critical"
    }
  ],
  "searchResultFixtures": [
    "examples/fixtures/search-results.json"
  ],
  "pageFixtures": [
    {
      "requestedUrl": "https://example.test/products/aurora-trail-shoe",
      "finalUrl": "https://example.test/products/aurora-trail-shoe",
      "htmlFixture": "examples/fixtures/product-page-clean.html",
      "statusCode": 200,
      "contentType": "text/html; charset=utf-8"

  ]
}
```
