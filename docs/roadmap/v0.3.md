# v0.3 Roadmap

## Roadmap intent

v0.3 builds on the completed v0.2 deterministic one-product analyzer.

v0.2 made the diagnostic core clearer by validating finding taxonomy cleanup, query-level visibility health, explicit URL evidence, focused HTML fixtures, and a minimal local scenario CLI.

v0.3 should make the analyzer more useful for realistic merchant-facing demos and future integrations without changing the core product boundary.

The theme for v0.3 is:

> Report usability and scenario realism.

The goal is not to add live data collection yet. The goal is to make local deterministic scenarios richer, reports easier to consume, and scenario authoring less error-prone.

## Progress tracking rules

This roadmap is the source of truth for v0.3 implementation progress.

Every phase must keep this progress block directly under the phase title:

```md

```

Allowed statuses:

- `Pending` — not started.
- `In Progress` — implementation has started but is incomplete or not fully validated.
- `Implemented` — code has been created, but acceptance criteria are not fully validated yet.
- `Validated` — implementation is complete and tests/acceptance criteria pass.
- `Blocked` — implementation cannot proceed until a dependency, design decision, or bug is resolved.
- `Skipped` — intentionally removed from v0.3 scope.

Progress rules:

- Use `0%` for `Pending`.
- Use an approximate percentage for `In Progress`.
- Use `80%` to `99%` for `Implemented` when tests or acceptance validation are still pending.
- Use `100%` only for `Validated`.
- Update `Implemented in` with commit SHA, branch name, or PR reference when available.
- Update `Validated by` with the exact test command, manual check, or acceptance evidence used.
- Do not mark a phase as `Validated` unless its acceptance criteria pass.
- Do not change the phase scope silently while marking progress. If scope changes, update the phase text explicitly.

Codex and future implementation chats should update these fields whenever a phase is started, implemented, blocked, skipped, or validated.

## v0.3 product boundary

The v0.3 core remains centered on this API shape:

```php
VisibilityAnalyzer::analyze(
    ProductSubject $product,
    array $queries
): VisibilityReport;
```

Where:

- `$product` is one product.
- `$queries` contains one or more query contexts for that product.
- Search-result evidence is still supplied from outside the core through a provider or scenario fixture.
- Page evidence is still supplied through a fetcher abstraction or scenario fixture.
- The analyzer returns one `VisibilityReport` for that product.

There is no `BatchVisibilityAnalyzer` in v0.3.

The minimal CLI remains scenario-driven:

```bash
bin/visibility analyze examples/scenarios/visible-clean.json
```

v0.3 may improve CLI output modes, scenario validation, and scenario coverage, but it must not turn the CLI into a crawler, scraper, SaaS worker, or batch monitoring tool.

## v0.3 non-goals

Do not implement these in v0.3:

- live Google scraping;
- live Bing scraping;
- marketplace scraping;
- Google Shopping scraping;
- AI answer-engine checks;
- embeddings;
- vector databases;
- semantic product-query matching;
- automatic expected-query generation;
- browser rendering;
- full-site crawling;
- Laravel integration;
- dashboard UI;
- SaaS account/project model;
- scheduled monitoring;
- competitor reports;
- batch analysis for many products;
- revenue forecasting or attribution modeling;
- external HTTP fetching as default CLI behavior.

## v0.3 design principles

- Keep the analyzer deterministic.
- Keep one-product analysis as the central unit.
- Prefer richer local scenarios over live external collection.
- Prefer small Codex-sized phases over broad feature batches.
- Preserve v0.2 status semantics: `visible`, `not_visible`, `uncertain`.
- Preserve v0.2 visibility health semantics: `healthy`, `at_risk`, `blocked`, `unknown`.
- Preserve v0.2 URL evidence roles: `matchedUrl`, `requestedUrl`, `finalUrl`, `expectedUrl`, `canonicalUrl`.
- Preserve existing JSON report compatibility unless a change is explicitly documented and tested.
- Make reports easier for humans to consume without removing structured evidence needed by machines.

## Phase 1 — Page transport and indexability scenarios

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** PR #30  
**Validated by:** composer test

### Goal

Add deterministic local scenarios for page transport and indexability failures that are already part of the MVP diagnostic promise.

### Problem

v0.2 validates clean, HTML noindex, canonical mismatch, and missing Product schema cases. The MVP success criteria also include HTTP failure, redirect/final URL problems, and `X-Robots-Tag` noindex, but these are not yet represented as focused CLI scenarios.

This phase should expand scenario coverage without mixing schema/content work into the same implementation prompt.

### Deliverables

Add deterministic local scenarios under `examples/scenarios/` and supporting fixtures under `examples/fixtures/` as needed for:

- product page HTTP error;
- product page redirect to a different product, category, or other non-equivalent URL;
- `X-Robots-Tag` noindex.

### Acceptance criteria

- Each new scenario is executable with `bin/visibility analyze <scenario-path>`.
- Each scenario isolates one primary diagnostic condition where practical.
- Each scenario has a test proving the expected finding code appears.
- Existing v0.2 scenarios continue to work.
- No scenario requires network calls, browser rendering, scraping, or external services.
- This phase does not add schema, offer, or content-alignment scenarios.
- This phase does not add compact output or merchant-facing summaries.
- `vendor/bin/phpunit` passes after implementation.

### Design notes

Prefer PageSnapshot-level fixture evidence for HTTP and redirect cases instead of creating misleading HTML-only fixtures.

For `X-Robots-Tag`, prefer explicit header evidence in the page fixture/scenario payload.

Do not implement batch scenario execution in this phase.

## Phase 3 — Compact report output mode

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** PR #32  
**Validated by:** composer test

### Goal

Add a compact report output mode so CLI demos can show the key diagnostic result without printing full page HTML, full parsed page evidence, and every nested evidence reference by default.

### Problem

The existing JSON report is useful as a complete machine-readable artifact, but it is too large for quick terminal inspection and merchant-facing demos. A clean scenario can produce a large JSON payload because it includes `pageSnapshot.body`, parsed page structures, and nested evidence.

The full report should remain available, but developers need a smaller view for quick diagnosis.

### Deliverables

- A compact output mode for the CLI, for example:

```bash
bin/visibility analyze examples/scenarios/visible-clean.json --compact
```

- A compact JSON shape that includes the essential information only:
  - product identity;
  - overall status;
  - overall priority;
  - query statuses;
  - visibility health;
  - top probable causes;
  - top recommended actions;
  - URL evidence summary;
  - warnings.
- Full output remains the default unless a different choice is explicitly justified.
- README documentation for compact output.

### Acceptance criteria

- Full JSON output remains available and backward compatible.
- Compact output omits full `pageSnapshot.body`.
- Compact output omits large parsed page internals unless needed for a top-level diagnostic.
- Compact output remains valid JSON.
- Compact output is deterministic and tested.
- CLI errors still go to stderr with non-zero exit codes.
- `vendor/bin/phpunit` passes after implementation.

### Design notes

Prefer implementing compact output as a serializer or report projector rather than mutating the core `VisibilityReport` object.

Do not remove evidence from the full report.

Do not introduce a non-JSON text format in this phase.

## Phase 4 — Scenario validation and authoring guardrails

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** PR #33  
**Validated by:** composer test

### Goal

Add deterministic scenario validation before CLI analysis execution so invalid local scenarios fail early with clear authoring feedback.

### Problem

Scenario JSON files are intentionally local and explicit, but small authoring mistakes can otherwise flow into analysis as missing or ambiguous fixture evidence. That makes demos harder to debug and can hide fixture mistakes behind analyzer output.

### Deliverables

Add deterministic validation before CLI analysis for common scenario authoring mistakes:

- missing product `expectedUrl`;
- missing or empty `queries`;
- missing search-result evidence for a query;
- invalid or missing local fixture paths;
- malformed page fixture definitions;
- duplicate or mismatched query/provider evidence where it creates ambiguity.

### Acceptance criteria

- Valid existing scenarios continue to run as before.
- Invalid scenarios fail before analysis.
- Invalid scenarios write clear error messages to stderr.
- Invalid scenarios return non-zero exit codes.
- Validation is deterministic.
- Validation does not require network calls, browser rendering, scraping, schema registries, external services, Laravel, databases, UI, or batch execution.
- Existing v0.2 scenarios and v0.3 Phase 1-3 scenarios continue to work.
- `composer test` passes for this phase.

### Design notes

Validation should catch clear authoring mistakes, not enforce merchant-business correctness.

Prefer a small validator near the CLI scenario loader.

Do not introduce JSON Schema dependencies unless clearly justified.

Do not change analyzer semantics, report serializers, finding codes, status semantics, `visibilityHealth` semantics, or scenario batch behavior.

## Phase 5 — Merchant-facing diagnostic summary fields

**Status:** Validated  
**Progress:** 100%  
**Implemented in:** PR #36  
**Validated by:** assumed composer test

### Goal

Add concise merchant-facing diagnostic summary fields to the report JSON so demos and future integrations can explain product invisibility in plain, actionable terms without relying on a dashboard or AI-generated prose.

### Problem

v0.3's theme is report usability and scenario realism. The final v0.3 phase should improve the report itself with deterministic merchant-facing diagnostic summary fields instead of replacing the intended product capability with documentation or demo workflow cleanup.

Existing structured findings and evidence are machine-readable, but demos and future integrations also need a stable, plain-language summary that helps merchants understand the primary issue and next step without requiring dashboard-specific presentation logic or generated prose.

### Deliverables

Add additive report JSON fields under `diagnosticSummary`, such as:

- `diagnosticSummary.title`
- `diagnosticSummary.statusExplanation`
- `diagnosticSummary.primaryIssue`
- `diagnosticSummary.merchantExplanation`
- `diagnosticSummary.recommendedNextStep`
- `diagnosticSummary.evidenceHighlights`

The summary fields must be:

- deterministic;
- derived from existing statuses, findings, priorities, recommendations, and evidence;
- concise and stable enough for merchant-facing demos;
- additive only, preserving existing JSON report compatibility;
- available alongside the existing structured findings and evidence.

### Acceptance criteria

- Summary fields are deterministic.
- Summary fields are derived only from existing statuses, findings, priorities, recommendations, and evidence.
- Summary fields do not require LLMs, AI APIs, embeddings, semantic matching, external services, live data collection, scraping, browser rendering, Laravel, SaaS, dashboard, batch analysis, or compact output changes.
- Existing structured findings and evidence remain available.
- Existing JSON report compatibility is preserved as an additive change.
- README documents the summary fields when Phase 5 is implemented.
- `composer test` passes after implementation.

### Design notes

Prefer an additive report serializer/projector or summarizer enhancement rather than mutating core analysis evidence.

Keep text concise, deterministic, and stable.

Do not attempt persuasive sales copy in the core package.

Do not add compact output changes; Phase 3 already owns compact output.

Documentation/demo cleanup can be deferred to v0.4 or a later documentation task.
